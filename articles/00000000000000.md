## 入力規則を考える
  記号の半角全角。数字は半角など
  https://ja.wikipedia.org/wiki/Wikipedia:表記ガイド
  https://andspace.net/blog/text_manual
  ### にくい と づらい
  ### 区切り線を入れる位置
  調査
  ### 複数の言葉を“と”でつなげる場合
  AとBとCのように3つの言葉が続く場合、「AとB、C」というように最初の2つは“と”で接続し3つ目以降の言葉は、
  “句点（、）”でつなく。
  ### “かつ”や“また”の句点を使用する場合
  句点の位置のルールはないとの事。
  ひらがなが続き、見にくい時や3つ事象が条件に並ぶ際などに「、かつ」を利用する。
  AとBをANDする際は、「AかつB」または「A、かつB」
  AとB、CをANDする際は、「A、BかつB」または「「A、B、かつB」

  ただし、各事柄を改行で表示で表現する場合、
    - “かつ＋改行”前の表現
      ```
      A
       and
      B
       and
      C
      ```
    - “かつ＋改行”後の表現
      ```
      A
      かつ、B
      かつ、C
      ```
      「、かつB」だと見にくいため
    
  ### 数字やアルファベット、記号は基本、半角数字
  ただし、一部記号「：（コロン）、－（ハイフン）、＋（プラス）、様々な括弧」などは、
  半角で表記し全体的なデザインが崩れてしまう際は、記号でも全角を使用する場合がある。
  ### 半角カナは使用しない
  ### 目を引きたいときなどはタイトルにも絵文字を入れる
  ### 本文で使用する絵文字は基本的にすべてのOSで表示できるものを採用する
  どうしても採用したい場合は、画像で表示する

## 変数名のデータ型を命名するハンガリアン記法（システムハンガリアン）
- メリット
  - 初学者・初級者がデータ型を意識して学習する事が可能となる
    プログラムのデータ型を意識してコーディングしないとバグの温床となってしまう。
    データ型を意識して命名する事で考えながらコーディングする事ができる。
    ただし、現在の主流（スタンダード）を事前に理解した上で学習する必要がある。
  - データ型が確認しにくいIDE[^1]を使用するレガシーな言語では開発しやすい
    [^1]: IDE（統合開発環境）：様々なツールが備わってるプログラム開発を支援するソフトウェア
    レガシーな言語のIDEでは、機能が乏しく現在ではアップデートも止まっており機能追加などもない。
    VS Codeを始めとする最新のIDEでは、デバッグ中にマウスオーバー[^2]する事で簡単に変数内の値やデータ型を確認する事ができるが、
    レガシー言語のIDEでは、デバッグ前にウォッチ式に追加するなど工程が必要で、尚且つ内容も確認しにくい。
    [^2]: マウスオーバー：マウスポインタを重ねる
  - 先頭文字を入力するだけで同じカテゴリのオブジェクトを一覧表示可能
    先頭の文字（プレフィックス）を入力すると変数名や関数・メソッド名が一覧表示されるので便利。
    ただ、命名規則がしっかりできれば、システムハンガリアン以外の方がメリットが大きい。
    例えば動詞を入力するだけで目的別の一覧が表示されるのでパッと見わかりやすい。
  - レガシーな言語の場合、予約語と変数名が被りにくい
    例えばVBAでは、アプリケーションに付属するプログラム言語となる為、予約語で利用されている名前が多い。
    名前が被りにくいよう導入するメリットがある。
  - 変数名や関数名・メソッド名が被りにくく、悩むことなく命名可能
    ただし、リリース後に発生しうるメンテナンスを考慮するとデメリットが多い。
    冗長な名前となる為、開発当初は小規模システムだったとしても機能の追加がくり返し行われ、
    最終的に大がかりなシステムとなった場合、保守性の悪いコードとなってしまう。
  - レガシーな言語におけるライブラリの命名規則と統一できる
    過去はシステムハンガリアンが推奨されていた事もあり、レガシーな言語のライブラリではシステムハンガリアン記法でめいめいされているので統一感がある。
  - VBAぐらいの小さな規模のプログラムでは、複数人で開発する事はまずないので、コーディング基準がなかれば好みの方を選択
- デメリット
  - 命名規則する為の命名規則が必要となり複雑化する
    大規模システム、かつ複数人の開発の場合、命名規則する為の命名規則が必要となってしまい複雑化する
    - 例えばExcel VBAの場合は下記ようなシステムハンガリアンの対応表をコーディング規則に作成が必要。
      - データ型：long → lngXXXX
      - データ型：String → strXXXX
      - データ型：配列［String()］ → aryXXXX
      - データ型：bool → blXXXX
  - 今後の展望を考慮した場合、システムハンガリアン以外で学んだ方がメリットが大きい
    初学者・初級者の場合に適正な名前で設計する事は難しいが、今後の展望を考えると最初にキャメル記法などで、
    悩みながら学んだ方が得られるメリットが大きい
  - データ型に変更があった場合、名前も合わせて変更する必要がある
    特に複数人で開発する大規模なシステムで採用した場合、
    仕様変更などによりデータ型の変更が発生してしまった際、影響範囲が大きい。

# データを“変更・修正・整形・加工・変換”の違い（個人的な見解を含む）
:::message
**個人的な見解を含む内容です。**

言葉自体の意味は辞書で調べていますが、最終的な結論は個人的な見解を含めてまとめています。
一つの意見としてご参考頂ければと思います。
:::
## Target
- 下記、IT用語の使い分けがあいまいな方
  - データ変更
  - データ変換
  - データ加工
  - データ修正
  - データ整形

## 辞書で調査
### それぞれの単語を調べてみる
#### 変更
  > 💡
  > 変え改めること。また、変わり改まること。変改。
  > 引用元: [コトバンク - 変更](https://kotobank.jp/word/変更-626201)
  > 
#### 変換
  > 💡
  > (━する) ある事柄・事態がそっくり別の事柄・事態に変わること。また、変えること。変更。
  > 引用元: [コトバンク - 変換](https://kotobank.jp/word/変換-8685)」
  > 
#### 加工
  > 💡
  > 人工を加えること。手を加えて細工すること。原料または他の製品に手を加えて、新しい製品を作ること。
  > 引用元: [コトバンク - 加工](https://kotobank.jp/word/加工-43977)
  > 
#### 修正
  > 💡
  > (━する) 文章、意見や計画、企画などの足りないところや誤り、不十分と思われる点を直し正すこと。しゅせい。
  > 引用元: [コトバンク - 修正](https://kotobank.jp/word/修正-182008)
  > 
#### 整形
  > 💡
  > (━する) 形を整えること。形を整えて、正常にすること。
  > 引用元: [コトバンク - 整形](https://kotobank.jp/word/整形-544908)
  > 

### 調べた結果のまとめ（個人的な見解を含む）
#### それぞれの意味合いを集約
| 言葉 | 意味合い |
| ---- | ---- |
| データ変更 | 大きく変化する際に使用。<br>変換や加工、修正、整形など、あらゆる手法を複数使用して大きく変化されること。 |
| データ変換 | 辞書では“変更”と同じ意味合い。<br>IT用語として使用する際の個人的な解釈は、「 変更[^1] > 変換[^2] 」という関係性で“変更”の方が少し大きなカテゴリーのイメージ。 |
| データ加工 | 加工前でも成り立っている状態に対して手を加えること。 |
| データ修正 | 修正前は誤り・不十分な点があり、それを直すこと。 |
| データ整形 | 整形前は不十分な点があり、それを直すこと。 |

[^1]: 例えば、テキストを開くエディターを サクラエディタ から VS Code に変化させる場合は、**変更**。
[^2]: 例えば、同じテキスト形式のファイル内で プレーンテキスト形式（\*.txt）から CSV形式（\*.csv） に変化させる場合は、**変換**。

#### それぞれの関係性
$$
変更 > 変換 > 加工 ≒ 修正 > 整形
$$

## 一次対応
何も操作せずリトライやディスクのクリーンナップ後に再処理など行うも変わらずエラーとなる
- バックアップ先の容量
  294GB / 465GB
- バックアップ元の容量
  | パーティション | 容量 | 使用済み | 未使用 |
  | ---- | ---- | ---- | ---- |
  | システムで予約済み | 50.00MB | 26.39MB | 23.61MB |
  | Cドライブ | 49.41GB | 30.24GB | 19.17GB |
  | 回復パーティション | 552.00MB | 465.72GB | 86.28MB |
  
## 原因
イベントビューアーのエラー内容を元に検索した結果、[こちら](https://kimassi.net/zak/?p=24529)の記事より
> いろいろ調べると、Windowsのシステムイメージを作る場合には、
> 500MB未満のボリュームに対して50MBの空き領域
> 500MBを超えるボリュームに対して320MBの空き領域
> 1GBを超える場合、各ボリュームについて、1GBの空き領域
> が必要という事がわかりました。

との内容を確認。

### 隠しパーティションの空き領域を確認する方法
Windows OSインストール時に自動的に作成される隠しパーティションのサイズは、
「ディスクの管理」画面で確認する事ができない。
- ディスク管理ツールを導入し確認する方法
  私は[MiniTool Partition Wizard](https://www.partitionwizard.jp/free-partition-manager.html)を導入し確認。
- PowerShellのコマンドレッド（Get-Volume）で確認する方法
  ```powershell:サンプル
  PS C:\Users\Administrator> get-volume

  DriveLetter FriendlyName       FileSystemType DriveType HealthStatus OperationalStatus SizeRemaining     Size
  ----------- ------------       -------------- --------- ------------ ----------------- -------------     ----
  D                              Unknown        CD-ROM    Healthy      Unknown                     0 B      0 B
              システムで予約済み NTFS           Fixed     Healthy      OK                     26.39 MB   50 MB
  C                              NTFS           Fixed     Healthy      OK                     19.17 GB 49.41 GB
                                NTFS           Fixed     Healthy      OK                     86.28 MB   552 MB
  ```
  - 参考記事
    https://atmarkit.itmedia.co.jp/ait/articles/1808/03/news031.html

“システムで予約済み”と“回復パーティション”の2つのボリュームで**容量の拡張が必要**である事が判明する。
| パーティション | 容量 | 使用済み | 未使用 | 対応有無 |
| ---- | ---- | ---- | ---- | ---- |

### なぜ、ミドルユーザー以上＋買い切り好きとしたのか
バックアップソフトを紹介する上で一番重要だと思ったポイントは、読者の対象をどうするかでした。

私なりに考えた結果、
- パソコンのミドルユーザー以上の方
- ソフトウェアを一度だけの買い切りで購入したい方

としました。
ここでは考えたユーザー層の定義を紹介します。
ユーザー層の定義等の細かい事が気にならない方は読み飛ばし次項をご参照ください。

※アコーディオンで最小化※
- オススメするユーザー層を考える
  バックアップソフトをオススメするユーザーを考察。まずユーザー層の定義は下記の通りとした。
  - ライト層（ライトユーザー）
    - メイン機がスマートフォン（スマホ）
    - そもそもデスクトップ型やノート型のパソコンを所有していない
    - パソコンを所有していたとしても、ほぼ利用していない
    - スマホでは、Apple iColudやGoogle Driveなどのオンラインストレージを使用
      （他の有名どころは、Microsoft OneDrive、Dropbox、Box、楽天ドライブ、Yahoo!ボックスなど）
    - パソコンを所有していた場合でも年賀はがきのデータや、アドレス帳など少量のファイルのみ保存
    
  このようなユーザーをライト層と定義した場合、保存しているデータのファイルサイズが小さい為、
  オンラインストレージの無料枠などで十分にバックアップ可能です。
  また、仮にパソコンが故障したとしても影響が少なく、
  バックアップソフトで対策するよりも中古や安価なノートパソコンやタブレットに買い替える方が、
  コストパフォーマンス（コスパ）が良いと考えました。

  このライトユーザーにバックアップソフトをオススメする場合、Windows OS用のバックアップソフトではなく、
  - 使用中のオンラインストレージの容量拡張
  - スマートフォン用のバックアップソフト

  のいずれかの方法が費用対効果があると感じます。

  - ミドル層（ミドルユーザー）
    - 少数のパソコンを所有しており頻繁に利用
    - スマホで撮った写真やビデオなど多くのファイルをパソコンに保存
    - ロースペックなNASを所有

  - ヘビー層（ヘビーユーザー）
    - 複数台のパソコンを所有しており頻繁に利用
    - ハイパフォーマンスなNASを所有
    - ミドル層のデータに加え、下記の大容量データをパソコンに保存
      - 個人製作物（グラフィック系）のデータ
      - 個人開発（プログラム系）のデータ

  以上のように定義した結果、**ミドルユーザー以上の方**をオススメとした。

- ミドルユーザー以上＋α（アルファ）の条件も考えた
  さらにオススメする環境を考えた結果、

  - 一度きりの買い切り型のソフトウェアが好き（サブスクが苦手）
  - NASや外付けHDDなどを複数台、所有（または、購入予定）
  - LANのネットワーク速度を意識して自宅に構築済み（または、構築予定）
  - 時間をかけて自宅のインフラを強化したい方

  という条件も考えた。

- バックアップソフトを導入・運用する際のデメリット
  ここまではバックアップソフトを導入するメリットに着目して条件に合うユーザーを考察したが、
  導入・運用する際のデメリットとして上げられる点は下記の通りと考えた。
  - バックアップ環境を実現するために構築や設定などの作業時間が必要
  - 構築時や運用中などトラブルが発生した場合、トラブルシューティングが複雑になりやすい
  - NASのハードウェアやHDDの故障などにより予期せぬ出費が発生する可能性も

## 類似ソフトとの比較
### 類似ソフトの定義
- フリーソフト版（無料版）が提供されていること
  体験版や試用版として期間の制限があるソフトは対象外
- フリーソフトをダウンロードするにあたり会員登録が不要であること
- 下記の機能がフリーソフトで実現できること
  - システムバックアップと復元
  - 保存先に共有フォルダが設定可能
- 公式サイト・マニュアルが日本語対応していること
- アップデートが頻繁に行われており現在も更新されていること

		→Windows標準のバックアップソフトと比較して有用性があるか。
		S: Serviceability = 保守性 or 運用性	障害復旧やメンテナンスのしやすさ
		→実際のソフトを操作して確認。また、Windows標準のバックアップソフトと比較。
		I: Integrity = 保全性 or 完全性	過負荷や障害、誤操作などによるデータの破壊や喪失、不整合などの起こりにくさ
		→誤操作に関しては、実際に操作して復元する際に、確認用のポップアップが表示されるか確認
		　喪失や不整合については、バックアップスケジュールなどを組んで結果を確認
		S: Security = 気密性	外部の攻撃者による不正侵入や遠隔操作、データや改ざんや機密データの漏洩などの起こりにくさ
		→独自のファイル形式で保存されるため、悪意がある人物により解析が難しい。また、LAN内（ホームネットワーク）を使った

	→実際にソフトを利用した結果で判断。比較元はWindows標準のバックアップソフト。
	・保守性
	完成したソフトウェアがどこまで修正しやすいかということです。作った本人にしか修正を行うことができないのであれば、改善が必要なときにコストがかかります。
	利用者に直接関係がない特性に見えますが、最終的なリリースまでにかかるコスト削減は、利用者へのメリットとなることがあるのです。
	→不明だが、企業がリリースしている専用ソフトである為、個人で作成したフリーソフトや、Windows標準のソフトよりは、予算や人員がかかっていると思われる。
	・移植性
	別の環境に移したいときに、どれほど容易に移せるのかという基準です。例えば、サーバーを移行するときやフレームワークを変更する際における、移行や変更のしやすさが重要になります。
	→システム要件を確認。また、バックアップデータそのものが別バージョンで機能するかなども評価対象。
	
	最近のPCはSSDであったり、クラウドを利用している事が前提の為、パソコンの全体のディスク容量は少ない。
	Windows標準のバックアップで保存先をネットワーク上のストレージにした場合、世代管理ができない為、不便
	
	－ライト: データ量: 少ない、予算: 少ない、保管パーツ: ない
	　　　　　Microsoft OneDrive、Google Drive、Apple iCloud、Dropbox、Box、楽天ドライブ、Yahoo!ボックス
	　　　　　→バックアップ対象が年賀はがきのデータや、連絡先ぐらいであればオンラインストレージ（無料枠）で事足りる
	－ミドル: データ量: 人による、予算: 普通、保管パーツ: 多少あり
			　ロースペックNAS環境や外付けHDD、過去PCのHDDなど
			　→バックアップ対象が写真やビデオ、音楽データ（CD取り込み）
	－ヘビー: データ量: 多い、予算: 多い、保管パーツ: 多い
			　ハイスペックNAS環境やサーバー機器
			　→ミドルの内容に加え個人制作物や個人開発データベースのデータなど
	
	- サブスクリプションが苦手
	- 外部ストレージ環境が既にある、または自身で構築したい方
	
	Macでは細かい設定はできないものの標準ソフトのime Machineで外部ストレージでも世代管理可能。

  保全の為、買い切りのバックアップソフトの購入を検討すると良い。
